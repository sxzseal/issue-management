# 用户故事与验收标准规范

主 skill 红线 8 只写了"每个用户故事必须有可测试的验收标准（Given/When/Then 格式）"，但没定义"可测试"的门槛，也没规定 §4 用户故事本身的结构。dev-dev 后续拆任务和写测试完全依赖 §4，一旦 AC 含糊，测试就变形式主义。本 enhancer 把可测试性、3C 原则、AC vs DoD 三个抓手落实到 user-value subagent 和主 skill 合成阶段。

## 必须遵守

- **每个用户故事采用 `As a <角色>, I want <目标> so that <价值>` 单句结构** — 缺"so that"（价值）直接不算合法故事。
- **每个用户故事至少 3 条 AC，Given/When/Then 三段齐全**（红线 8 复述 + 细化）。
- **每条 AC 必须满足"可测试三要素"**：
  1. **可观察**：结果是外部观察者能看到的（UI 出现某状态、接口返回某字段、数据库某记录被创建），不是"感觉更好"。
  2. **有明确输入**：Given 段列出前置状态和触发数据，不能是"用户在页面上"这种空 setup。
  3. **有明确通过条件**：Then 段是布尔或枚举可判定的，不允许"较快 / 合理 / 良好体验"。
- **AC 与 DoD 分离**：
  - AC = 这个具体 story 的验收（在 §4 里）
  - DoD = 全项目质量线（测试覆盖率、a11y、i18n、性能 budget），进 §6 非功能需求 —— 不允许把 DoD 塞进单个 story 的 AC 里。
- **每个故事必须挂到 §5 的功能条目**（`FR-<id>`），反向 §5 每条功能至少被 1 个故事覆盖 —— 主 skill 合成阶段做交叉核验，缺失记 synthesis-log。
- **优先级承接** —— 故事优先级和它挂载的 §5 功能保持一致；若不同优先级挂同一故事，取更高优先级。

## 3C 原则（Card / Conversation / Confirmation）

写完故事必须能回答这 3 个问题；缺任何一个走 P1 问题（见 [[scoring-rubric]]）。

| C | 是什么 | 落在哪 |
|---|--------|--------|
| **Card** | 一句话故事本体（As a ... I want ... so that ...） | §4 每个故事的第一行 |
| **Conversation** | 需要澄清的细节 / 边界 / 假设 | §4 故事下 "Notes" / §10 开放问题 |
| **Confirmation** | AC（Given/When/Then） | §4 故事下 "AC" 段 |

## 故事条目标准模板

```markdown
### US-<id>: <一句话主题>

**故事**：作为 <角色>，我想要 <目标>，以便 <价值>。

**关联功能**：FR-<id>, FR-<id>
**优先级**：P0 / P1 / P2

**Notes**（可选）：
- <前置假设 / 与其他 story 的依赖 / 已知边界>

**AC**：
- **AC-1**：
  - Given <前置状态：数据/权限/入口>
  - When <触发动作：点击/提交/超时>
  - Then <预期结果：UI 状态 / 数据变化 / API 返回>
- **AC-2**： ...
- **AC-3**： ...
```

## 可测试 AC 示范 vs 反面

### ✅ 可测试

```
Given 用户已登录且购物车包含 3 件商品
When 用户点击"结算"按钮
Then 页面跳转到 /checkout，且订单摘要区显示 3 件商品的名称、单价、总价
And POST /api/orders 被调用，请求体包含 3 个 items
```

### ❌ 不可测试（改写要求）

| 反例 | 问题 | 改写方向 |
|------|------|---------|
| "响应速度较快" | "较快"无阈值 | Then 服务端响应 ≤ 300ms（p95） |
| "用户体验流畅" | 主观，无观察点 | Then 加载态在 200ms 内消失，无闪烁 |
| "支持所有主流浏览器" | 无法枚举 | 挪进 §6 非功能，AC 只写具体动作 |
| "确保数据安全" | 无判定条件 | Then 未授权请求返回 401，且 /api 日志记录被拒 IP |

## AC vs DoD 边界

放 §4 单个 story AC 里的：
- 该 story 独有的功能路径判定
- 该 story 明确涉及的错误分支（如登录失败提示）

放 §6 非功能需求（DoD 性质）：
- 全站 a11y（WCAG AA / 键盘导航）
- 全站 i18n（中英切换 / RTL）
- 全站性能 budget（LCP / TBT）
- 全站测试覆盖率（≥ 80%）
- 全站错误处理规范

## 主 skill 合成期核验清单（Step 2.5.4）

从 3 份 draft 融合出 §4 后，主 skill 亲自跑一遍：

1. **格式核验**：每个故事有 As/I want/So that 三段？缺任一 → P0，回填或补写。
2. **AC 核验**：每故事 ≥ 3 条 AC？每条 Given/When/Then 齐全？发现空 setup 或含糊 Then → P1，重写。
3. **可测试三要素核验**：随机抽 3 条 AC 手写测试草稿，写不出来的 → P0，返回打磨。
4. **FR 交叉核验**：`for each US: assert exists FR linked`，`for each FR: assert exists US covering it` —— 双向覆盖，缺失记 synthesis-log。
5. **DoD 越权核验**：搜 §4 AC 里出现"测试覆盖率 / 性能 / 兼容性 / a11y"等词 —— 命中即挪到 §6。

## 推荐模式

- **优先级 vs AC 密度**：P0 故事至少 4 条 AC（含 1 条异常路径 + 1 条边界），P1 至少 3 条，P2 至少 2 条。
- **异常路径的 AC 独立成条**：不要塞进正常路径的 Then 里；单独一条 AC 写"When 网络错误 / When 权限不足 / When 数据为空"。
- **可以引用 §7 API**：AC 里出现 API 时用 `POST /api/orders` 这种可跳转格式，别写"调用后端"这种含糊话。

## 反模式（禁止）

- ❌ AC 只有 Then 没有 Given/When — 无法在测试里 setup
- ❌ AC 用"应该"、"通常"、"一般"这种概率词 — 无法判定通过条件
- ❌ AC 里出现"用户满意 / 体验好 / 更快" — 主观判断，不可测试
- ❌ 用户故事没有 so that 价值 — 只有功能没有理由，dev-dev 判断不了取舍
- ❌ 一个故事塞 10+ 条 AC — 拆成多个故事，一个故事解决一个用户目标
- ❌ 把 §6 DoD 项塞进 §4 AC — 会导致每个 story 都得测 a11y，测试爆炸
