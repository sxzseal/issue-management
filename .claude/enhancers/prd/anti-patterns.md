# PRD 反模式清单

评分卡（[[scoring-rubric]]）给分数，本 enhancer 给一份"典型问题识别指纹"—— 主 skill 合成后和每轮打磨前先跑一遍 checklist，命中即写入本轮的 P0/P1/P2 待办。反模式来自 Atlassian PRD 最佳实践的浓缩：PRD 不是签字盖章的一次性文档，而是持续对齐的活文档。

## 必须遵守

- **Step 2.5.4 合成完成后**：主 skill 逐条扫下方 checklist，命中的写进 synthesis-log.md "反模式扫描" 段，并按 P0/P1/P2 定级（映射规则见下）。
- **Step 5.5 每轮打磨前**：至少扫结构类 + 内容类两组，锁定章节里的问题也要扫（用户想 lock 但里面还藏着反模式的话，AskUserQuestion 提示）。
- **P0 反模式（结构性缺失）阻断定稿**：命中即写入待办，用户不能"忽略"直接定稿（可以选择"降级为 P1 并写入 §10 开放问题"，但要显式确认）。

## 结构类反模式（触发即 P0）

| 反模式 | 识别信号 | 修法 |
|--------|---------|------|
| **无"不做"清单** | §9 空 / 只写"暂无" | 强制回到 scope-confirmed.md 抄"不做"组过来 |
| **无成功指标** | §2 指标是"提升体验 / 增加用户"这种非量化描述 | 补 SMART 指标或明确标 `TBD metric` + 敲定 owner |
| **无用户故事价值** | §4 故事缺 "so that" 段 | 逐条回填价值，见 [[acceptance-criteria]] |
| **AC 不可测试** | §4 AC 含"较快/合理/流畅/满意"等主观词 | 见 [[acceptance-criteria]] 可测试三要素 |
| **单一作者视角** | 通篇看不到设计/工程输入的痕迹（如无技术风险、无交互细节讨论） | 提示：dev-prd 走 3 视角 fan-out 就是为解决这个；若仍单一，重跑 refine-view |
| **PRD 变实现规格** | 出现"用 React 组件 xxx"、"数据库表结构 yyy"这种实现细节 | 把实现细节挪到 dev-dev 的 task-state / 技术设计文档，PRD 只写行为契约 |

## 内容类反模式（触发即 P1）

| 反模式 | 识别信号 | 修法 |
|--------|---------|------|
| **假设当事实** | §1 背景 / §3 用户画像出现未经调研的绝对化陈述（"用户都希望…"） | 挪到独立 §"Assumptions" 段，标"待验证" + owner |
| **决策藏在散文里** | 关键决策（如"付费墙放在第 3 步"）淹没在段落中间 | 提到 §"Decisions" / 章节末尾的 Decision Log |
| **开放问题伪装成已完成** | §5 功能里出现"待定 / TBD / 也许"但没进 §10 | 全部挪到 §10 开放问题 + 挂 owner + 下个检查点 |
| **术语不一致** | 同一概念多个名字（如"会员/订阅用户/付费用户"混用） | 首次出现时锚定，后续统一；主 skill 合成期就该规范化 |
| **竞品分析只有名字** | §1/§9 提到竞品但无定价/功能对比/差异化 | 回到 research-brief 补数据；若数据没查到，标"待补" |
| **优先级空标签** | 功能标了 P0 但没说为什么 P0（无理由） | 每个 P0 补一句"因为 <场景/指标/依赖>"，红线 7 复述 |

## 过程类反模式（触发即 P2）

| 反模式 | 识别信号 | 修法 |
|--------|---------|------|
| **文档过长** | §5 功能列表 > 30 条，单个 story 描述 > 15 行 | 拆多个 loop / 按 P0/P1/P2 分组，MVP 只保 P0 |
| **信息重复** | 同一约束在 §2 / §6 / §7 多处出现且措辞不同 | 单点定义 + 引用 |
| **AC 密度失衡** | 有些 story 8 条 AC 有些 story 1 条 | 见 [[acceptance-criteria]] "优先级 vs AC 密度" |
| **调研引用缺失** | PRD 里的具体断言（"用户平均等待 5s"）没有 `[R<n>]` 来源 | 补 research-brief 引用 或 标"内部估算"；红线 2 复述 |
| **过度设计** | 出现"未来可能会 / 为将来扩展预留 / 灵活支持 xxx" | MVP 阶段砍掉，写"当前不支持"，避免设计吸引开发投入 |

## 反模式 → P0/P1/P2 映射规则

- 结构类命中 → P0（阻断定稿）
- 内容类命中 → P1（应当解决，可协商）
- 过程类命中 → P2（打磨优化，成本低就顺手改）

若同一问题多类命中 → 取最高级别（结构类为准）。

## 扫描输出格式（synthesis-log.md 追加 / 每轮 iteration.note）

```markdown
## 反模式扫描 — v<N>

### 结构类（P0）
- ❌ 无"不做"清单 — §9 只写"暂无"，回填 scope-confirmed.md 的"不做"组
- （其他）

### 内容类（P1）
- ⚠️ §5 FR-004 标 P0 但未给理由 — 补一句"因为对应核心场景 US-2"

### 过程类（P2）
- 💡 §1 提到竞品 Pomotroid 但无功能对比 — 打磨时补一段表格
```

## 推荐模式

- **合成后主动扫，不等用户催**：Step 2.5.4 合成完立刻跑一遍，写进 synthesis-log；Step 5.5 展示 PRD 摘要给用户前也扫一次，让用户看到你已经自查过。
- **锁章节前扫一遍**：用户想 lock 章节时，先扫该章节，命中反模式的必须解锁问题后才允许 lock。
- **反模式 + 评分卡结合使用**：反模式扫描告诉主 skill "哪里错了"，评分卡给出"错到什么程度"，二者结合决定是否需要再打磨。

## 反模式（元反模式）

- ❌ 扫完不落盘 — 主 skill 心里知道有问题但没告诉用户 = 违背红线 6
- ❌ 只扫一次就再也不扫 — 每轮打磨都要扫，前一轮修的问题可能被引入新问题
- ❌ 扫出问题不分级 — 直接堆一大坨列表，用户不知道先改什么
- ❌ 把过程类当结构类阻断定稿 — P2 只是建议，不能因为"文档略长"就不给定稿
