# 分类型追问框架

主 skill Step 1 的 4 维通用澄清（目标 / 范围 / 约束 / 成功指标）能圈出大方向，但每种产品类型都有自己特有的盲区（ToC 缺留存策略、ToB 缺权限体系、平台缺冷启动、AI 缺兜底）。本 enhancer 给主 skill 一份"看菜下饭"的追问清单：先判类型，再选对应清单往需求里补 3-5 个针对性问题。

## 必须遵守

- **Step 1 澄清完通用 4 维后**，主 skill 必须先判定项目类型（下表五类，取一个主要类型；混合型选最主导的一个），再从对应"分类追问"里挑 3-5 条**没被通用问答覆盖**的问题追问用户。
- **单轮问题上限 5 条**：每轮 AskUserQuestion 别一次塞超过 5 个（3 类分类问 + 通用 4 类里没澄清的 = 一批），避免用户被淹没。
- **用户已经说过的信息不重复问**（红线 6 复述）。
- **类型判定结果写到 research-brief.md 顶部**（`> 项目类型：<ToC | ToB | Tool | Platform | AI-Product> / 主导 + <次要类型>`），后续 3 视角 subagent 都能看到。
- **主 skill 不代替用户拍板**：追问是引出信息，答案必须是用户说的（红线 11 复述）。

## 类型判定（一句话锁定）

| 类型 | 判定信号 |
|------|---------|
| **ToC** | 面向个人消费者、下载/注册即可用、社交/娱乐/生活服务 |
| **ToB** | 面向组织/团队、有账号体系或权限、按席位/规模计费 |
| **Tool** | 单机或轻量协作、核心是"输入 → 处理 → 输出"pipeline（编辑器/生成器/转换器） |
| **Platform** | 双边或多边市场，连接供给方与需求方（内容/交易/服务撮合） |
| **AI-Product** | AI 能力是核心卖点（不是辅助），输出质量决定产品价值 |

**混合判定**：如 SaaS 里做 AI Copilot → 主 ToB，次 AI；开源工具站里带社区 → 主 Tool，次 Platform。主导类型走主清单，次要类型挑 1-2 个高相关问题补充。

## 分类型追问清单

### ToC 追加问题

| # | 问题 | 目的 |
|---|------|------|
| C1 | 用户从哪来？（搜索/社交/口碑/买量） | 决定 onboarding 长度、首屏内容 |
| C2 | 使用频率是每天/每周/偶尔？ | 决定留存策略与通知设计 |
| C3 | 有没有社交/分享属性？病毒系数期望多少？ | 决定是否要做邀请码/分享链路 |
| C4 | 需要哪些敏感权限（位置/通讯录/相机/支付）？ | 决定隐私合规和授权时机 |
| C5 | 是否有游客模式？注册墙放在哪一步？ | 决定首次体验流畅度 |

**PM 主动建议章节**：新用户引导、消息通知策略、用户成长体系（如适用）、隐私合规声明位置。

### ToB 追加问题

| # | 问题 | 目的 |
|---|------|------|
| B1 | 目标企业规模？个人 / 中小 / 大企业？ | 决定架构复杂度和定价档 |
| B2 | 使用者 = 决策者吗？（决策链条几层） | 决定销售策略与试用设计 |
| B3 | 需要多角色/权限吗？（Owner / Admin / Member / Guest） | 决定账户体系设计 |
| B4 | 要跟哪些现有系统集成？（SSO / CRM / ERP / Slack） | 决定 API 出口 |
| B5 | 合规要求？（GDPR / 等保 / SOC2 / HIPAA） | 决定架构和数据存放 |
| B6 | 数据导出/迁移诉求强吗？ | 决定"退出成本"和信任建设 |

**PM 主动建议章节**：账户体系（组织→团队→角色→用户）、审计日志、SLA、客户成功流程。

### 工具类追加问题

| # | 问题 | 目的 |
|---|------|------|
| T1 | 核心 pipeline 是什么？（输入 → 处理 → 输出，一句话说完） | 锁定 MVP 骨架 |
| T2 | 输入数据来源和格式？（本地文件 / URL / 剪贴板 / API） | 决定接入方式 |
| T3 | 输出格式和落地渠道？（下载 / 复制 / 云盘 / API 回调） | 决定成品交付方式 |
| T4 | 要不要历史记录/版本管理？ | 决定存储设计 |
| T5 | 有性能敏感操作吗？（实时 / 大文件 / 高并发） | 决定技术选型 |
| T6 | 单用户 or 需要协作？（同时编辑？评论？） | 决定协作能力边界 |

**PM 主动建议章节**：快捷键、批量操作、导入导出格式、模板 / 预设、撤销重做栈。

### 平台类追加问题

| # | 问题 | 目的 |
|---|------|------|
| P1 | 连接的是哪两方？（供给方 vs 需求方） | 锁定双边用户模型 |
| P2 | 冷启动策略？先攒供给还是先攒需求？ | 决定 MVP 侧重 |
| P3 | 平台抽成/收费模型？（成交抽 / 订阅 / 广告 / 免费） | 决定交易流程 |
| P4 | 信任机制？（评价 / 认证 / 担保 / 保险） | 决定信任体系 |
| P5 | 是否需要内容审核？（人工 / 机审 / 举报） | 决定运营后台复杂度 |
| P6 | 纠纷处理机制？（客服介入 / 平台仲裁 / 用户自决） | 决定售后 SOP |

**PM 主动建议章节**：双端界面差异、匹配/推荐算法、交易保障、纠纷流程、供给方入驻门槛。

### AI 产品追加问题

| # | 问题 | 目的 |
|---|------|------|
| A1 | AI 是核心还是辅助？（Copilot / Autopilot / Feature） | 定位 AI 在体验中的权重 |
| A2 | 用户对 AI 输出质量的容忍度？（可以试错 / 不能出错 / 需人复核） | 决定人工兜底方案 |
| A3 | 是否收集用户反馈来优化？（点赞点踩 / A-B / 显式评分） | 决定反馈闭环设计 |
| A4 | 调用量和成本预期？月活/日均调用/单次成本上限？ | 决定定价和配额 |
| A5 | 数据合规和用户数据训练？（用户数据是否入训练集） | 决定隐私政策边界 |
| A6 | 失败/幻觉时如何降级？ | 决定错误体验 |

**PM 主动建议章节**：AI 输出可控性参数（温度/长度/风格）、失败兜底、用量配额、反馈机制、免责声明。

## 类型判定 + 追问输出格式

判定完把结果和用户回答落到调研简报顶部，让 3 个 subagent 都能读到：

```markdown
# 调研简报：<需求主题>

> 项目类型：ToB SaaS（主导）+ AI-Product（次要，AI 是核心 Copilot 能力）
> 澄清补充：
> - B3: 需要 Owner/Admin/Member 三级权限
> - B4: 需要 SSO（SAML）+ Slack Webhook
> - A2: 客服场景，人工必审，AI 建议先让人 review
> - A4: 预计 1w 席位 × 每日 20 次调用，单次成本上限 $0.02
...
```

## 推荐模式

- **判定不确定时用 AskUserQuestion 让用户选**：给用户看这 5 类的一句话定义，让 TA 挑最像的一个 + 是否有次要类型。
- **混合类型合并两份清单挑高优先级问题**：如 ToB + AI，先问 B1/B3/B5 + A2/A4/A6，其他放弃。
- **通用问题里已隐含答案的直接跳过**：如用户在通用 Q4"成功指标"里已经说 DAU，就不用再问 C2 频率了。

## 反模式（禁止）

- ❌ 一次抛出全部 20+ 分类问题 — 用户会 lost；单轮最多 5 条
- ❌ 判定完不落盘 — 类型判定必须写进调研简报，否则 3 个 subagent 拿不到
- ❌ 忽略混合型只按单类问 — SaaS AI 产品只问 ToB 不问 AI，会漏兜底策略
- ❌ 主 skill 替用户选类型不确认 — 类型影响 PRD 主线，含糊时必须 AskUserQuestion
- ❌ 追问变审讯 — 每问一个都要短暂告诉用户"这个信息用来定 xx 章节"，让用户理解为什么问
